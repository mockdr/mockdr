# ADR-005: Deterministic Seed Data with Fixed RNG and Dependency-Ordered Seeding

**Status**: Accepted
**Date**: 2026-03-11

## Context

Integration tests must produce the same assertions on every run. If seed data is randomly generated with a different seed each time, tests that assert on specific field values (agent IDs, threat classifications, user tokens) will be non-deterministic. Equally, tests that rely on relational integrity (e.g. a threat's `agentId` references a real agent) require seeding to respect dependency order.

## Decision

Seed data is generated by `backend/infrastructure/seed.py` calling per-domain seeder modules under `backend/infrastructure/seeders/`. Every run begins with:

```python
def generate_all() -> None:
    random.seed(42)
    fake = Faker()
    Faker.seed(42)
    store.clear_all()
    reset_registry()
```

Seeder modules are called in strict dependency order, each returning IDs for downstream seeders:

```python
account_id, account_name = seed_account(fake)
user_ids               = seed_users(fake, account_id, account_name)
site_ids               = seed_sites(fake, account_id, account_name)
group_ids_by_site      = seed_groups(fake, account_id, account_name, site_ids)
agent_ids              = seed_agents(fake, account_id, account_name, group_ids_by_site)
seed_threats(fake, agent_ids)
seed_alerts(fake, agent_ids)
# ... remaining domains
```

**Reference data** (names, paths, OS variants, hardware models) is defined as static constants in `backend/infrastructure/seeders/_shared.py`, not generated by Faker — ensuring debuggability:

```python
_MALWARE_NAMES = ["Ransom.WannaCry", "Trojan.Emotet", ...]   # 24 real malware names
_OS_VARIANTS   = [("Windows 10 Pro", "windows", ...), ...]   # 9 tuples
_AD_DOMAIN     = "DC=acmecorp,DC=internal"                   # fictional org
```

All domain names use the fictional `acmecorp` namespace. No real organisation names or infrastructure details appear in seed data.

**Seed counts** are configurable via environment variables (e.g. `SEED_COUNT_AGENTS=60`) with defaults, allowing test suites to run with a smaller dataset for speed.

## Alternatives Considered

| Option | Rejected because |
|---|---|
| Random seed each run | Non-deterministic tests; flaky CI |
| Hardcoded JSON fixtures | No relational integrity; hard to scale; breaks when IDs change format |
| Database migrations + fixtures | Overkill for an in-memory mock; adds infrastructure dependency |
| Faker without fixed seed | Produces different names/values each run; assertions cannot be stable |

## Consequences

- **Positive**: Every test run produces the same entity graph — IDs, names, field values are stable across runs and machines
- **Positive**: Failing tests on CI are reproducible locally with no environment variables needed
- **Positive**: Reference data (malware names, OS variants) is human-readable in source, not buried in generated output
- **Negative**: Seed data reflects a snapshot in time; new fields added to domain objects require updating the corresponding seeder — a deliberate, reviewable change
- **Negative**: `random.seed(42)` means all seeder modules share a single RNG sequence; reordering seeders or adding a seeder changes downstream values (treated as a known trade-off, not a bug)
